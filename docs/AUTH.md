# Authentication

**Status:** current

Email/password authentication via [better-auth](https://www.better-auth.com),
with per-user data isolation. Signup can be gated (invite code and/or email
allowlist); each account only ever sees its own products and entries.

## Pieces

- `src/lib/auth.ts` ‚Äî better-auth server config (Drizzle/Neon adapter,
  email+password, min length 8, Resend-backed `sendResetPassword`).
- `src/app/api/auth/[...all]/route.ts` ‚Äî the better-auth request handler.
- `src/lib/auth-client.ts` ‚Äî React client (`signIn`, `signUp`, `signOut`,
  `requestPasswordReset`, `resetPassword`, `changePassword`).
- `src/lib/auth-helpers.ts` ‚Äî `getSessionUser()` and `requireUserId()` for
  server components and actions.
- `src/proxy.ts` ‚Äî redirects unauthenticated users to `/login` and
  authenticated users away from the auth pages (optimistic cookie check). This
  is the Next.js 16 successor to `middleware.ts`; the exported function is named
  `proxy`.
- `src/db/auth-schema.ts` ‚Äî `user` / `session` / `account` / `verification`
  tables (generated by the better-auth CLI), re-exported from `db/schema.ts`.

## Routes

- `(auth)` group (public): `/login`, `/signup`, `/forgot-password`,
  `/reset-password` ‚Äî centered layout, no app chrome.
- `(app)` group (protected): dashboard, `/products`, `/entries`, `/account` ‚Äî
  the layout fetches the session and renders the header + user menu.

## Data isolation

`products.userId` references `user.id` (cascade delete). Entries and expenses
inherit ownership through their product. **Every** server action calls
`requireUserId()` and scopes its queries:

- product reads/writes filter or set `userId`;
- record create/update/delete verify the target product is owned by the user
  (returning a friendly error otherwise);
- dashboard and entries queries inner-join products filtered by `userId`.

Defense in depth: the proxy gates routes, the `(app)` layout re-checks the
session, and the actions re-check ownership ‚Äî so a forged request still can't
read or mutate another user's data.

## Signup gating

A `hooks.before` middleware in `src/lib/auth.ts` gates the `/sign-up/email`
endpoint:

- **`SIGNUP_INVITE_CODE`** ‚Äî when set, the signup form must submit a matching
  invite code. The form shows an "Invite code" field automatically (via the
  `signupRequiresInviteCode` flag).
- **`SIGNUP_ALLOWED_EMAILS`** ‚Äî when set, the signup email must appear in this
  comma-separated allowlist.

Both can be combined (both must pass). If neither env var is set, signup is
open ‚Äî set at least one for a personal deployment. Sign-in, password reset, and
change-password are unaffected.

## Password flows

- **Forgot:** `/forgot-password` ‚Üí `requestPasswordReset({ redirectTo:
  "/reset-password" })` ‚Üí better-auth emails a tokenized link via Resend.
- **Reset:** the link lands on `/reset-password?token=‚Ä¶` ‚Üí `resetPassword`.
- **Change:** `/account` ‚Üí `changePassword` (current + new, revokes other
  sessions).

If `RESEND_API_KEY` is unset, the reset link is logged to the server console so
the flow still works in development.

## Environment

See [`.env.example`](../.env.example): `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`,
`RESEND_API_KEY`, `EMAIL_FROM`, and the optional signup gates
`SIGNUP_INVITE_CODE` / `SIGNUP_ALLOWED_EMAILS`. Auth tables are created by
`pnpm db:push`.
